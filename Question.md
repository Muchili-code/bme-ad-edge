# 针对边缘计算辅助诊断“黑盒”展示形式的分析与建议

## 一、结合论文对 App 构建流程的检查

在重新核对论文原文、完整工作流以及《展示形式.md》后，当前展示逻辑有四个必须讲清楚的点：

1. **“低分辨 dMRI 和 T1 直接送入模型”不能讲得太粗糙**
   - 论文里的 DisC-Diff 确实利用 T1 作为解剖先验，但工程展示时必须明确：PC 端已经完成 dMRI 与 T1 的预配准，板端拿到的是统一坐标空间下的数据包，而不是两份彼此毫无空间关系的原始图像。

2. **超分后到 RI 之间，必须显式补上 DTI 拟合和 V1 提取**
   - 从超分后的 dMRI 到 RI，中间不是直接点积。
   - 正确链路是：超分后 4D dMRI -> DTI 张量拟合 -> 提取主扩散特征向量 V1 -> 与皮层表面法向量点积 -> 得到 RI。
   - 这一步不是 DisC-Diff 模型内部完成的，而是超分之后的后处理步骤。

3. **展示主线要围绕 RI，而不是把 ADC、皮层厚度等内容都塞进去**
   - 论文的核心创新点是 RI、RI Skewness、RI Maximum。
   - ADC、皮层厚度可以作为补充背景，但不适合放到初版 Demo 的主叙事里，否则会冲淡“dMRI 超分后做皮层柱层向分析”的亮点。

4. **前端演示不能说成“跑一张 2D 图”**
   - dMRI 是 3D/4D 体数据。
   - 因此界面上更专业的说法应是“局部 3D patch 或代表性切片的超分流程演示”。
   - 这个 patch 只是为了可视化模型的局部计算思路，不代表论文只对单张切片做超分。

## 二、我需要科研负责人给我提供哪些数据？

因为你要在终端本地预存 HC、MCI、AD 三组典型病例，所以需要科研负责人提供每组至少 1 个已脱敏病例的数据包。建议数据分成四类：

1. **展示数据**
   - 原始低分辨 dMRI 的代表性切片与局部 patch
   - 原始 T1 的代表性切片与局部 patch
   - 超分后 dMRI 的代表性切片与局部 patch

2. **板端继续计算所需核心输入**
   - 全脑超分后的 4D dMRI
   - 对应的 `bvec/bval`
   - dMRI 与 T1 的预配准结果说明
   - 皮层表面法向量
   - 11 个等间距深度点的三维坐标
   - ROI 映射关系和采样索引

3. **T1 表面与几何先验**
   - FreeSurfer 生成的 pial surface、white surface
   - ROI 分区结果

4. **参考模板与报告样式**
   - 典型 ROI 的 RI 深度曲线模板
   - 全脑 74 个分区的参考统计范围
   - 报告模板字段

这里的关键变化是：

- **不再要求 PC 端预先拟合好 V1**
- **而是要求 PC 端提供足够让板端本地拟合 V1 的输入**

也就是说，PC 端给的是“超分后 4D dMRI + bvec/bval + 几何先验”，不是“已经算完的 V1 最终答案”。

## 三、哪些步骤在 PC 上完成，哪些步骤在板端完成？

这是当前工作流里最核心的分工表：

| 处理步骤 | 运行位置 | 理由/说明 |
| :--- | :--- | :--- |
| dMRI 预处理（去 Gibbs、畸变校正、eddy current、bias field 等） | PC 端 | 依赖 MRtrix3/FSL/ANTs 等重型工具，不适合板端现场执行。 |
| T1 结构像处理与表面重建 | PC 端 | FreeSurfer 耗时长、内存重、流程复杂，必须提前完成。 |
| dMRI 与 T1 的预配准 | PC 端 | 这是空间统一的前提，属于科研端离线准备工作。 |
| 全脑 DisC-Diff 超分 | PC 端 | 当前比赛版本不在板端真实运行完整扩散模型。 |
| 局部超分流程演示 | 板端前端 | 只做流程可视化，不做真实 DisC-Diff 推理。 |
| DTI 张量拟合 | 板端后端 | 这是“板端自己算 V1”的第一步，基于超分后 4D dMRI 与 `bvec/bval` 本地完成。 |
| V1 提取 | 板端后端 | 由板端从本地 DTI 拟合结果中提取，不再依赖 PC 二次工作。 |
| RI 计算 | 板端后端 | 用板端拟合出的 V1 与法向量做点积，得到 RI。 |
| RI 曲线、ROI 指标、异常脑区、报告生成 | 板端 | 这是你要向评委强调的“板端后续全部闭环”。 |

一句话总结：

**PC 端负责把重型科研流程一次性做完并打包；板端在接收标准化数据包后，本地完成 DTI -> V1 -> RI -> 报告，不需要 PC 再次参与。**

## 四、当前方案最容易被质疑的点，以及如何提前堵住

### 漏洞 1：评委质疑“你们板端根本没跑超分模型”

**回应策略：**

> 当前比赛版本是工程展示原型。板端不真实运行完整 DisC-Diff，全脑超分由 PC/工作站端离线完成；板端展示的是局部 patch 的超分流程可视化，并在此之后继续本地完成 DTI 拟合、V1 提取、RI 计算和报告生成。

也就是说，超分这一段是“演示型”，但超分之后的链路是“板端继续计算型”。

这样讲比“全部都在板端实时完成”更稳，也更不容易被追问击穿。

### 漏洞 2：评委质疑“既然板端自己算 V1，为什么文档里又说不能加载 4D dMRI？”

这个质疑是对的，所以文档必须统一成下面这套说法：

> 不让**前端**直接把完整 4D dMRI 全量载入内存，不等于不让**板端后端**读取 4D dMRI 计算包。

正确表述应该是：

- 前端只展示轻量图片、图表和结果 JSON
- 板端后端按需或分块读取本地 SSD 中的超分后 4D dMRI
- 在本地完成 DTI 拟合和 V1 提取

因此，“板端自己算 V1”与“前端不能整包加载 4D 大文件”并不冲突。

### 漏洞 3：评委质疑“你们是不是还要让 PC 再干活？”

必须坚持这句：

> PC 端在赛前一次性完成预处理、预配准和全脑超分，并将标准化数据包交付给板端；板端拿到数据包后，后续 DTI 拟合、V1 提取、RI 计算和报告生成全部在本地完成，不再依赖 PC 再次参与。

这句是你整个展示策略的逻辑核心，不能松。

### 漏洞 4：评委质疑“板端内存会不会炸”

回应方式不要说“我们不算了”，而要说：

- 不把完整 4D dMRI 交给前端
- 把 4D dMRI 放在本地高速存储中
- 由板端后端按需读取、分块处理
- 前端只接收 V1 采样结果、RI 曲线、ROI 指标和报告 JSON

所以你们要强调的是**数据访问方式优化**，不是回避板端计算。

---

## 五、现阶段最稳妥的答辩口径

推荐这样说：

> 我们的系统采用 PC/科研端与边缘终端协同架构。PC 端一次性完成 FreeSurfer、dMRI 预处理、预配准和全脑 DisC-Diff 超分，并把超分后 4D dMRI、bvec/bval 以及皮层几何先验打包到设备中。边缘终端现场不真实运行 DisC-Diff，而是展示局部超分流程；随后在本地继续完成 DTI 拟合、V1 提取、RI 计算和报告生成。这样既保证了现场演示的稳定性，又满足“PC 处理一次，板端独立完成后续流程”的目标。

不推荐这样说：

- “我们在板端实时运行完整全脑 DisC-Diff。”
- “我们在板端实时完成 FreeSurfer、全脑配准和全部科研流程。”
- “板端只负责读结果，不做真正计算。”

第一种会被算力追问打穿，第二种会被流程真实性打穿，第三种则会削弱你现在最重要的核心逻辑。
